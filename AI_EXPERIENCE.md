# AI 开发经验总结 (AI_EXPERIENCE.md)

## UI/UX 优化经验
- **东方矿物色系的按钮与标签体系**: 针对浅色与半透明的“斋房/原料”页面，采用两态矿物色设计：
  - **未选中态**: 半透明留白，加细微边框混以微量墨色 (`rgba(60,59,54,0.15)`)，字体颜色为深色 (`rgba(60,59,54,0.8)`), 背景带有明显的 `backdrop-filter: blur` 以吸收底部色彩。
  - **选中态**: 深厚底蕴（如 `linear-gradient(135deg, rgba(60,59,54,0.90) 0%, rgba(40,39,34,0.95) 100%)` 配搭暗金或浅杏色文本 `#f7f0e4`），摒弃多余的随机多色彩映射，保持极简禅意。

- **东方美学风格指南**: 在处理具有“水墨”、“禅意”调性的项目时，应避免使用过于鲜艳或现代化的纯色背景。优先选择带有磨砂玻璃（Backdrop Filter）和渐变（Gradient）的材质，配色倾向于竹青、黛蓝、象牙白。
- **文案意境感**: 负面情绪引导不应仅停留在功能性词汇（如“抱抱”），应上升至哲学高度（如“万般心绪，皆是过客”），能显著提升产品的品牌质感。

- **东方红色系运用**: 在酒类或热情主题中，使用“赭石”或“茶褐”代替高饱和度的纯红（Red-500），能更好地融入禅意背景，同时保持深邃感。
- **字态重构 (Typography)**: 对于传统韵味的 UI，使用 `serif`（宋体/楷体）类字体并辅以较宽的 `letterSpacing` (0.05em)，能瞬间拔高视觉的“历史感”与“庄重感”。
- **沉浸式详情页 (Immersive Detail Refactoring)**: 
  - **分层视差**: 使用毛玻璃容器 (`backdrop-filter`) 覆盖在带有“水墨蒙版” (`mask-image`) 的大幅饮品图片上，创造出极强的纵深感。
  - **汉字逻辑化**: 在具备东方美学的应用中，将步骤编号（一、二、三）进行汉字重构，比阿拉伯数字更能传达品牌调性。
  - **悬浮交互岛 (Floating Action Bar)**: 对于全屏详情，底部应采用悬浮式玻璃岛设计，并利用 `env(safe-area-inset-bottom)` 实现跨设备高端适配。
### 3. 数据一致性规范 (Data Consistency)
- **多处计数统一**: 涉及原料总数统计时，**必须**统一采用 `new Set()` 结合 `name_cn || name` 进行去重。严禁在不同组件中使用不一致的过滤/统计逻辑（如某一处去重而另一处不去重）。

### 4. 东方美学视觉规范 (Oriental Aesthetics)
- **字体应用**: 核心文本（标题、饮品名、副标题、标签、语录）应全量覆盖宋体/楷体 (`"Songti SC", "STKaiti", "KaiTi", serif`)。
- **强制翻译策略**: 针对外部 API 数据，必须在前端展示层强制通过 `translateDrinkName` 或 `translateIngredient` 字典进行中文化。即使 API 返回英文，也需优先匹配本地字典或 `name_cn` 字段。
- **版式限制**: 侧重横向滚动 (`flex-nowrap`) 的单行版式，通过 `overflow-x-auto` 保持界面的呼吸感，避免传统的列表堆叠。英文残留（如 VODKA, GIN）会极大破坏禅意 UI 的协调性。

### 5. 性能监控与透明度 (Performance & Transparency)
- **流程时间线 (Timeline)**: 针对长链路逻辑（如多 Agent 联合推荐），必须在控制台输出带时间戳 (`performance.now()`) 的结构化日志。
- **原子化埋点**: 日志应区分“开始”、“步骤内耗时”和“渲染就绪完成点”。
- **LLM 吞吐与延迟优化 (Throughput & Latency)**: 
    - **模型选型策略**: 对于语义蒸馏 (Semantic Distillation) 类任务，7B 级别模型通常比 72B 级别快 5-10 倍，且精准度差异在关键词提取场景下可接受。
    - **Prompt 压缩**: 将嵌套型 Schema 压缩为扁平结构，并移除下游未使用的冗余字段，可显著减少生成 Token 量，进而降低延迟。
    - **端到端流式 (Streaming)**: 使用 SSE (Server-Sent Events) 同步转发大模型 Token，前端配合 `ReadableStream` 实时消费，能极大提升用户对系统的响应感。

## 组件治理
- **会话级状态隔离 (Session Isolation)**: 处理“初始同步自全局但允许局部独立修改”的需求时，使用 `useRef` + `useEffect` 构建“一次性初始化”锁是非常稳健的方案。这能有效防止 React 重渲染或依赖项变动导致的“状态回溯”现象，实现真正的会话独立性。
- **全局计数一致性 (Data Normalization)**: 当系统中存在 ID 唯一但业务逻辑要求名称唯一（如原料）的情况时，必须在所有入口和展示层实施统一的去重逻辑（如 `new Set()`）。忽视展示层的去重会导致 65 vs 66 这种极度干扰用户的 UI 冲突。
- **UI 同步显现优化 (UI Sync Stabilization)**: 在实现堆叠式画廊或多卡片展示时，应避免使用过大的百分比偏移（如 `70%`），因为这可能导致卡片在窄屏下初始位于离屏区域。应使用较小的稳定偏移（如 `12%`）或固定像素。同时，必须为基于 Ref 的宽度计算提供非零默认值，以防止 React 初次挂载时因 `containerWidth` 为 0 导致 `NaN` 偏移而产生的视觉抖动或延迟。
### 4. 推荐语个性化与批次差异化 (Batch Personalization)
- **经验**：在 LLM 生成批量文案时，模型容易“偷懒”生成雷同句子。必须在 Prompt 中加入明确的批次序号 (Batch Index) 并要求不同的切入角度。
- **意象化**：传统的古诗词引用在现代 UI 中容易显得突兀且重复。通过建立五行到视觉意象（如木-松林，火-暗火）的映射表，并要求“画面感叙事”，可以显著提升文案质量。
- **本地降级**：本地 fallback 逻辑必须使用 `drinkName` 等唯一标识作为随机种子（Hash Seed），防止同一批次内所有卡片显示相同的兜底文案。

### 5. 函数签名与作用域安全 (Scope Safety)
- **教训**：在重构逻辑时（如在 `philosophyTags.js` 中增加 `drinkName` 依赖），务必同步检查所有调用栈。仅在底层函数使用新变量而未在中间层级传递会导致 `ReferenceError`，进而引发整个渲染组件崩溃。
- **工具限制**：浏览器自动化工具在输入中文字符时可能存在限制。在验证国际化应用时，优先准备等效的英文 Prompt 或通过修改代码注入测试数据，而非盲目重试。

## 推荐语优化经验 (2026-03-13)

### 1. 异步文案的“视觉无缝”策略
**问题**：当 LLM 异步生成的文案替换本地降级文案时，若格式不一（如括弧、长度突变），会产生明显的“闪跳”感。
**解决**：将本地降级文案的样式（括弧 「」、三段式结构）调整到与 LLM 预期输出完全一致。即使内容发生替换，由于视觉轮廓重合，用户感知极低。

### 2. LLM 提示词颗粒度与描写深度
**经验**：单纯限制字数（如 15-28 字）会导致 AI 偏向生成干瘪的逻辑句。通过要求“增加画面感描述”（如冷热体感、生活化类比）并放宽字数到 25-45 字，可以显著提升文案的档次感，使其从“机器拼接”转向“酒保管怀”。

### 3. 文案生成的“一次性稳定”与原子性锁
**经验**：在多 Agent 流水线中，如果多个阶段都可能触发文案润色，必须引入 `useRef` (或等效状态锁) 确保全局仅触发一次生成任务。
**策略**：核心计算（如向量搜索）完成后立即触发异步生成，并使用锁变量（如 `isQuoteFetching`）屏蔽后续流程的冗余补偿调用。这能有效消除因重复刷新导致的 UI 文案多次跳变现象。

### 3. 环境热重载失效处理
**教训**：在修改 `server/llmProxy.js` 等后端代码时，如果发现 Prompt 调整未生效，应第一时间手动检查进程并重启。
## UI 布局与视觉增强经验 (2026-03-13)

### 1. 卡片布局的“重心偏移”策略
**经验**：在以图片为背景的卡片设计中，通过 `flex-col justify-end` 结合 `inset-0` 的布局，可以将视觉重心压低。移除固定的 `pb` (padding-bottom) 并使用 `mt-auto` (margin-top: auto) 强制将非核心信息（如原料列表）推至底部，能有效解决内容较少时卡片中间“悬空”导致的不美观。

### 2. 文本信息的“视觉层级”构建
**策略**：
- **标题间距**：加大酒名与下方内容的 `mb` (margin-bottom)，为复杂的背景图提供“视觉呼吸区”。
- **装饰线锚点**：在长段推荐语左侧增加单像素或双像素的 `border-l` (左边框) 和 `pl` (左内边距)，不仅能起到视觉引用符的作用，还能在复杂的彩色背景中通过线条引导用户的阅读视线。
- **字态提升**：将推荐语从 `text-xs` (12px) 提升至 `text-[14px]`，配合 `font-medium` 和 `leading-relaxed`。在 14px 这个临界点上，系统字体的“渲染质感”通常优于 12px，能显著提升文案的“说服力”与醒目度。

### 3. 事件穿透与交互安全
**经验**：在使用 `absolute inset-0` 构建装饰性容器时，必须严防 `pointer-events` 导致的点击失效。模式应为：外层容器 `pointer-events-none`（穿透），内层内容块 `pointer-events-auto`（拦截），确保背景装饰不影响卡片本身的点击跳转。

### 4. CSS 兼容性与 Linter 优化 (2026-03-13)
- **指令替代方案**: 当 IDE 或 Linter 无法识别 Tailwind 的 `@tailwind` 指令（报 Unknown at rule 警告）时，推荐使用标准 CSS 的 `@import` 语法：
  - `@import "tailwindcss/base";`
  - `@import "tailwindcss/components";`
  - `@import "tailwindcss/utilities";`
  这不仅能消除警告，还符合标准的 CSS 模块化规范，且 Tailwind 官方同样支持。
- **多端兼容属性**: 在使用类似 `-webkit-line-clamp` 的私有属性实现多行文本截断时，**必须**同步定义标准的 `line-clamp`属性，以确保在不同浏览器内核（如 Firefox/Safari）及其更新版本中的稳健表现。

## AI Agent 系统集成经验 (2026-03-13)

### 1. 领域专家 Agent 的解耦与聚合
**经验**：针对特定垂直领域（如调酒学）的 AI 功能，应创建专门的领域专家 Agent（如 `MixologyExpert`），而非将其逻辑散落在各个 UI 组件或通用的分析器中。
**优势**：通过统一的代理接口（如 `executeMixologyTask`），可以标准化输入校验输出验证、错误处理以及降级逻辑，使 UI 层保持极致简洁（KISS原则）。

### 2. 异步导入与按需加载
**策略**：在多 Agent 系统中，由于 Agent 类及其依赖可能较大，推荐在执行辅助函数（Helper Functions）内部使用显式的 `await import()` 进行异步加载。
**结果**：这能显著降低首屏 JS Bundle 大小，同时在需要时才触发 LLM 相关模块的初始化。

### 3. 多模式 Agent 的统一上下文
**经验**：一个 Agent 可以支持多种模式（如 `ANALYZE` 和 `ASSIST`）。通过 `AgentContext` 传递 `taskType`，可以在同一个 Agent 类中复用相关的私有方法和辅助逻辑，同时保持代码的紧凑性。
